# 竞品与灵感研究方法

更新时间：2026-07-22（Asia/Shanghai）

## 目标与边界

本知识库服务于 `2048-all-in-one` 的产品、表现与 GF Framework 使用决策。研究对象覆盖 2048、合并解谜、网格肉鸽、回合制小型策略与具有突出表现设计的精品独立游戏。研究只提炼机制、设计模式与可验证事实，不复制第三方代码、Shader、音频、图像或其他素材。

默认只改动 `docs/research/indie_game_benchmark/`。游戏代码与 `addons/gf` 不在研究批次的修改范围内。若证据指向 GF 缺口，先记录反馈候选；任何框架改动仍需独立 issue、分支、PR 与版本回流。

## 来源优先级

| 等级 | 可接受来源 | 用途 |
| --- | --- | --- |
| A | 固定 commit 的原始仓库、仓库 LICENSE、作者/发行商实机视频的可复核时间戳 | 架构、实现事实、许可证、动效/音效/UI 观察 |
| B | 作者 itch.io 页面、官网、官方开发日志、Steam 开发者公告或商店页 | 产品功能、平台、发布日期、作者对设计的说明 |
| C | 媒体、Wiki、聚合页或社区讨论，且至少由一个 A/B 来源交叉验证 | 补充检索与历史语境 |
| D | 由画面或源码作出的推断 | 只作为待验证假设，必须明确标注“推断” |

搜索结果页、自动摘要、仓库 topic 和视频标题本身不作为关键结论的唯一证据。来源失效、区域限制或无法观察视频内容时，报告必须降低证据等级，不能补造事实或时间戳。

## 计数口径

- **候选**：唯一游戏/项目名称，至少附一个作者、发行商或原仓库链接，并说明与本项目的相关维度。
- **深度分析**：独立报告完整覆盖证据版本、玩法、表现、音频、UI/UX、功能、性能线索、风险、当前项目差异和 GF 映射；未知项明确写“未证实”。
- **实机视频**：可实际观察的 gameplay、playthrough、demo 或开发日志视频；每条记录上传方、链接、观察日期和至少一个可复核时间戳。纯预告片只有在出现连续实际操作画面时才计数。
- **行动建议**：必须同时包含外部证据、当前项目基线/缺口、建议所有者、GF 复用或缺口判断、验收条件和优先级。一个笼统方向不能拆成多个重复计数。

## 第三方仓库处理

唯一允许的下载根目录为：

```text
E:\_workspace\Godot Project\_research\2048-benchmark
```

规则如下：

1. 只执行获取与只读检查，例如 `git clone`、`git rev-parse HEAD`、`git show`、`git ls-tree`、文本搜索与读取。
2. 不运行、构建、导入或安装第三方项目；不初始化 submodule；不执行仓库脚本、二进制、宏或包管理器生命周期命令。
3. 每个报告记录远端 URL、分析 commit 的完整 SHA、抓取日期、许可证文件路径与许可证结论。未找到许可证时按“保留所有权利/不可复用”处理。
4. 仓库、视频缓存与素材不进入本项目，不纳入 Git，也不作为运行时依赖。
5. 宽松许可证也不自动授权素材复用；代码许可证、素材许可证、商标和第三方依赖分别判断。

## 当前项目比较口径

基线以当前仓库提交与声明为准：

- 项目意图与约束：[`.gf/project_contract.json`](../../../.gf/project_contract.json)
- 架构和所有权：[架构说明](../../architecture.md)
- 视觉方向：[视觉规范](../../visual_style.md)
- 已有能力和维护路线：[README](../../../README.md)、[roadmap](../../roadmap.md)
- 模式目录：[game_mode_registry.tres](../../../features/gameplay/resources/registries/game_mode_registry.tres)

比较时使用四种结论：

- **可借鉴**：当前缺少或深度不足，且能在项目边界内实现。
- **已有/深化**：机制已存在，应改善内容、参数或产品表达，不再造平行系统。
- **不适配**：会破坏确定性、可读性、平台预算、纸媒视觉方向或产品定位。
- **框架反馈候选**：多个项目都需要的通用机制，且已排除项目业务规则、供应商差异和本地策略；只记录，不直接修改 `addons/gf`。

## GF 映射规则

先从能力意图查找已安装 GF 9.0.1，再核对精确 API 与源码职责。GF 只拥有通用机制，项目拥有模式规则、内容语义、平衡、视觉层级、产品流程和平台策略。

常用映射边界：

| 需求 | 现有 GF 机制 | 项目仍应拥有 |
| --- | --- | --- |
| 回合/变体编排 | `GFTurnFlowSystem`、`GFActionQueueSystem`、确定性随机 | 规则、敌人/障碍语义、奖励和平衡 |
| 动效与 Shader | `GFShaderParameterUtility`、`GFRenderWarmupUtility`、命名视觉队列 | Shader 内容、反馈层级、降级策略与预算 |
| 音频 | `GFAudioUtility`、`GFAudioEvent`、`GFAudioBank` | 事件命名、素材、混音和自适应音乐规则 |
| 输入与可访问性 | `GFInputMappingUtility`、焦点/文本适配工具 | 手势语义、减少动态、高对比、教程和可访问性产品策略 |
| 存档/挑战 | `GFSaveGraphUtility`、`GFSeedUtility`、平台 Adapter 边界 | 每日种子规则、签名、赛季与排行榜权威性 |
| 诊断/性能 | `GFOperationDiagnosticsUtility`、有界诊断 | 项目指标、采样点、隐私和验收预算 |

若能力目录搜索无结果，但精确 API 已存在，记为“目录/发现性缺口”，而不是重复实现运行时机制。

## 优先级

- **P0**：直接影响核心操作反馈、可读性、首次体验、确定性或平台性能预算；应进入最近一个验证切片。
- **P1**：显著提升留存、内容深度、可访问性或展示价值；P0 基线稳定后推进。
- **P2**：有价值的扩展、实验或长尾内容；先做低成本原型和指标门槛。

## 批次提交安全

每个研究批次只精确暂存本目录。提交前检查 `git status --short` 和 `git diff --cached --name-only`，确保没有第三方仓库、素材、生成缓存或用户已有改动进入提交。推送后再进入下一批。
