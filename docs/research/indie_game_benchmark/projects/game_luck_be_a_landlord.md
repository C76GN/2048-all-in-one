# Luck be a Landlord 深度产品分析

研究日期：2026-07-22（Asia/Shanghai）
类型：老虎机符号构筑 / 经济压力肉鸽
证据：开发者官网、itch.io、官方开发日志与开发者上传实机视频；未下载或运行游戏。

## 产品与核心循环

玩家每轮旋转老虎机，用产出的金币按周期交房租；每次旋转后选择新符号，符号间相互消费、邻接、转化或增益，逐渐形成 build。官方 itch 页明确说明没有真钱赌博或微交易，并列出大量符号、物品、随机局与约半小时会话。

实际实机 [00:42](https://www.youtube.com/watch?v=JcbKTeJ6uXU&t=42s) 显示 5 列转轮在大面积橙色背景上运行，水果符号使用粗黑轮廓，左下金币 `27`、底部单一 SPIN 主按钮始终可见；决策与反馈被压缩在一个稳定屏幕内。

## 表现：特效、Shader、动效

- 粗轮廓、纯色底、低分辨率符号和机械滚轴是可读性优先的“廉价机器”美术语义；没有证据表明核心依赖复杂 Shader。
- 转轮动效把多个收益分步结算，适合突出连锁来源；当前观察帧未显示完整结算序列，不能量化节奏。
- 对本项目最有价值的是“每个符号关系可逐项解释”，而不是模仿老虎机外观。

## 音效

官方页链接原声与声音作者，但当前未独立试听，无法确认转轮、掉币、稀有连锁的实际分层。产品语义天然适合按收益/稀有度逐级音高与并发限制；这是建议，不是已验证实现事实。

## UI/UX 与可访问性

- 主操作只有 SPIN，符号选择与物品选择在回合间出现，避免持续占用棋盘。
- itch 页明确提供屏幕阅读器支持：可把当前高亮文本输出给操作系统，并建议键盘/手柄；同时声明色盲友好、可配置控制、blind friendly。
- 官方移动端日志记录玩家要求横屏，并展示 portrait/landscape 都需可用的产品考虑。当前项目尚缺屏幕阅读语义、高对比/色觉策略，这是明确领先点。

## 玩法变体与功能设计

深度来自符号池与物品池的组合爆炸、周期租金压力和“是否继续稀释池子”的选择。它证明即使主操作单一，也可通过可组合内容形成长期策略。内容数量巨大同时带来 tooltip、检索和因果解释负担。

## 性能线索

itch 页显示 Godot 制作，桌面包约 90–107 MB，并覆盖 Windows/macOS/Linux、键鼠/手柄和多语言。官方未公开帧时间或内存；大量符号联动意味着结算应批处理并限制表现并发，但这是架构推断（D），不能据此断言其具体实现。

## 与当前项目比较及 GF 映射

| 发现 | 当前项目判断 | GF / 所有者 |
| --- | --- | --- |
| 单一动作 + 高密度符号组合 | 当前规则多但局内选择少；可借鉴内容构筑 | 确定性 seed/turn flow；符号池、权重与经济归项目 |
| 周期房租形成短期目标 | 当前目标主要是达到数值 | progress/挑战 section 可用 SaveGraph；目标节奏归 gameplay |
| 屏幕阅读器与色盲声明 | 当前确认缺完整可访问性 | 输入、焦点、文本/设置机制可复用；产品语义归项目；GF accessibility 仍是发现性候选 |
| 分步连锁结算 | 当前 Action Queue 与语义音频底座合适 | 复用命名 Action Queue、`GFAudioEvent`，增加跳过/加速/减少动态 |

## 可借鉴点、风险与证据限制

P0 建立方块/按钮/状态的可访问文本投影和减少动态选项；P1 试做“小型符号池 + 周期目标”的确定性挑战模式。不要复制符号图、美术、音频或具体交互表。大量内容若没有搜索、图鉴与因果回放会恶化理解成本；当前项目已有图鉴/回放，应把它们作为解释层。
