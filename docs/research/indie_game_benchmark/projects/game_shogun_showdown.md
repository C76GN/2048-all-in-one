# Shogun Showdown 深度产品分析

研究日期：2026-07-22（Asia/Shanghai）
类型：单轴站位 / 攻击编排回合肉鸽
证据：开发者 itch.io、发行商官网与发行商上传的官方实机视频；未下载或运行游戏。

## 产品与核心循环

Shogun Showdown 把回合战术压缩到一条横向战线：移动、转向、装填攻击牌或执行已排好的攻击都会推进节奏，敌我站位和朝向决定结果。官方页面强调“每个行动都重要”、位置与攻击时机、升级并组合 attack tiles、获得技能与构筑牌组，以及死亡后的新角色、攻击和技能解锁。

官方发行日视频在 [00:42](https://www.youtube.com/watch?v=Y1sJu_B1NFY&t=42s) 展示实际场景：玩家进入火把照明的洞窟/首领空间，中央石像、角色白色聚光和背景剪影形成清楚焦点，底部只留下绿色分段计量。该帧能证明场景入场与氛围层次，但不能单独证明攻击队列的具体交互，相关机制以官方产品文字为据。

## 表现：特效、Shader、动效

- 日本风像素美术以剪影、局部暖光和大面积暗部建立舞台感；角色所在位置用高亮而非大量 HUD 包围。
- 单轴布局允许攻击位移、命中停顿和队列执行成为主要动效，不需要频繁移动镜头。对本项目的价值是“保持棋盘参照系稳定，让反馈沿规则方向运动”。
- 观察帧中的火把、聚光和洞窟层次可能由精灵、灯光或 Shader 组合，视频不足以区分实现方式，也无法量化粒子、过渡或屏幕震动。

## 音效

本轮没有独立试听音轨，官方功能文字也没有给出混音或事件系统细节，因此不评价打击、拔刀、环境或音乐。若借鉴“预排后连续执行”，本项目应利用已有语义音频事件为入队、确认、逐段命中、连锁完成设不同层级，并限制同帧并发。

## UI/UX

- 核心空间只有一轴，玩家可以把注意力用于朝向、距离和攻击次序；这是通过删减自由度换取更深可读性。
- 攻击牌既是能力又是待执行动作，必须清楚区分“已装填、下一步执行、冷却/不可用”。官方页面能证明组合 attack tiles 的设计，但本轮单一观察帧没有覆盖这些具体状态。
- 入场画面弱化控件、强化角色与目标，适合作为节奏间隔；实际回合仍应保持危险提示，不可让氛围遮蔽规则。

## 玩法变体与功能设计

升级攻击牌、组合执行次序、角色技能与解锁池共用同一套单轴战斗语法。其深度来自“先安排、后结算”的时序，而不是地图复杂度。对 2048 项目可迁移为一到三步的可视行动队列，或在特定挑战中预选两个方向后统一结算；但必须保留一步滑动的即时性，避免把所有模式改造成牌组游戏。

## 性能线索

小型像素角色、单轴战场和受控同屏单位数看似有利于性能，但这只是画面推断（D）。官方页面未公布帧率、内存、加载、平台差异或低端档数据。队列式连续攻击可能在同一结算窗口触发多段位移、VFX、音频与震动；如做原型，应记录队列峰值、取消/跳过动画后的终态和 `<50 ms` 输入反馈，而不是只看平均帧率。

## 与当前项目比较及 GF 映射

| 发现 | 当前项目判断 | GF / 所有者 |
| --- | --- | --- |
| 单轴站位与朝向形成深度 | 当前支持稀疏/自定义棋盘，但规则仍以全盘滑动为主 | 棋盘/模式资源可表达拓扑；朝向与敌人规则属于 gameplay，不扩 GF |
| 攻击牌先编排后执行 | 当前有命名 Action Queue，但它是表现/动作生命周期，不是玩家卡牌系统 | 复用 `GFActionQueueSystem` 编排结算；队列内容、合法性和平衡归项目 |
| 角色、攻击、技能逐步解锁 | 当前有图鉴、成就和进度，可承载展示，缺局内 build | 规范持久化走 SaveGraph；解锁经济与内容属于项目 |
| 舞台式局部光效 | 当前 CMYK 主题已有 Shader Profile 与预热 | 作为主题反馈 Profile 试验；经 `GFShaderParameterUtility`，同时提供减少动态/低档 |
| 行动时机高度可读 | 当前反馈丰富但缺敌方节奏/意图 | 回合顺序复用 `GFTurnFlowSystem`；意图提示和教程归项目 UI/gameplay |

## 可借鉴点、风险与证据限制

P1 可做一个不含卡牌收藏的“二步预排挑战”：界面先显示方向队列，再按确定性规则结算，且撤销、书签和回放完全一致。风险是队列 UI 与现有一步滑动手感冲突，也可能误用 GF Action Queue 存业务真相；业务状态必须由 gameplay 数据拥有，Action Queue 只负责编排与生命周期。不要复制角色、攻击牌、图像、音频或数值。本报告对攻击 UI 的描述来自官方文字，观察时间点只直接支持入场氛围。
