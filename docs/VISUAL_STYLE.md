# 视觉风格指南

本文档记录 `2048-all-in-one` 的视觉方向。后续修改背景、方块、菜单、HUD、弹层、动效或主题资源时，先对齐本文档，再改具体资源和脚本。

## 视觉 Thesis

目标气质：柔和肌理扁平独立游戏。

关键词：

- textured flat design
- noise illustration style
- soft indie game polish
- quiet geometric clarity
- low-stimulation pop color

这不是像素风。画面基础应是清晰的现代扁平图形，颗粒和肌理只是轻量覆盖，不能变成粗颗粒、马赛克、脏雾或视觉噪音。

## 总体原则

1. 先保证结构，再添加风格。
   - 菜单、HUD、设置、暂停、游戏结束和模式选择必须先有稳定布局、清晰层级和可读文字。
   - 如果去掉 shader、颗粒和动效后界面仍然难读，问题在布局，不在装饰。

2. 保持低刺激。
   - 默认状态要安静，强颜色只用于选中、焦点、重要反馈和高价值方块。
   - 不要让背景、按钮、方块和粒子同时抢注意力。

3. 少用高能装饰。
   - glow、blur、强渐变、强阴影、扫描线都必须克制。
   - 发光不能成为默认层级表达，深度主要靠明度、间距和边缘定义。

4. 资源是视觉真相来源。
   - 方块颜色来自 `resources/themes/tile_schemes/*.tres`。
   - 棋盘色来自 `resources/themes/board/*.tres`。
   - 表现层不应硬编码覆盖主题资源，除非是明确的临时调试或测试态。

## 背景

背景使用 `resources/shaders/game_background.gdshader`。方向是深色、柔和、带轻微有机流动的肌理背景。

推荐约束：

- `base_color` 使用深冷中性色，不使用纯黑。
- `secondary_color` 和 `accent_color` 只做低对比色带，不要高饱和大面积冲撞。
- `warm_color` 用作少量温度，不要把画面推成棕橙主题。
- `grain_strength` 建议保持在 `0.008` 到 `0.020`。
- `stipple_strength` 建议保持在 `0.000` 到 `0.006`。
- `scanline_strength` 建议保持在 `0.000` 到 `0.008`。
- `glow_strength` 建议保持在 `0.000` 到 `0.100`。
- `pulse_speed` 建议保持在 `0.000` 到 `0.080`。

禁止方向：

- 粗块状颗粒、方格化噪点、像素化马赛克。
- 大面积霓虹渐变。
- 背景亮度高到压过棋盘。
- 高速流动或明显呼吸闪烁。

## 色彩系统

项目采用深色背景 + 柔和高价值色块的系统。颜色可以有独立游戏的个性，但默认状态必须留有呼吸。

中性色角色：

- 背景：深冷色，负责沉静和空间。
- 面板：半透明深色，负责分组，不负责装饰。
- 文本：暖白或低饱和浅色，保证中文可读。
- 次级文本：低对比青灰或暖灰，不与主文字同权重。
- 边缘：低透明度暖色或冷色，只用于区分层级。

强调色角色：

- 青绿：交互 hover、可用状态、生成反馈。
- 暖金：焦点、关键提示、合并反馈。
- 玫红/酒红：pressed、危险或怪物相关反馈。
- 蓝紫：特殊模式、空间层次或少量强调。

避免：

- 一个界面被单一色相统治。
- 默认按钮大量使用高饱和填充。
- 红、黄、蓝、绿同时满强度出现。
- 用纯白、纯黑或 raw neon 当作主要 UI 色。

## 方块与棋盘

方块是游戏第一视觉信号。它们必须比背景更清晰，但不能刺眼。

规则：

- 方块保持扁平实心色块，不使用阴影和描边作为默认造型。
- 方块数值层级必须通过色阶可读，低值柔和，高值更有能量。
- 字体颜色必须优先保证对比度，不能为了色板统一牺牲可读性。
- 棋盘空格要比方块明显更低对比，避免空格抢占视觉。
- 方块颜色不得在 `GameBoardController`、`Tile` 或动画 Action 中随意覆盖主题资源。

主题资源建议：

- `classic_tile_theme.tres` 保持最稳、最容易读，是视觉基准。
- `fibonacci_tile_theme.tres` 和 `lucas_tile_theme.tres` 可以更明亮，但应避免纯黄、纯蓝、纯紫长时间大面积刺激。
- `red_tile_theme.tres` 和 `blue_tile_theme.tres` 用于特殊类型或模式时，需要和背景保持成熟对比，不要变成警示色海报。

## UI 面板与控件

UI 应服务重复游玩，不做 landing page。

布局：

- 主菜单、模式选择、设置、暂停、回放、书签和游戏结束都应使用一致的间距、字号和控制密度。
- 1280x720 是基础验证尺寸；1920x1080 和窄屏需要重新检查，不允许文字重叠、裁切或溢出。
- 面板之间用空间和标题层级区分，少用嵌套卡片。
- 目标达成弹层不是强制结算页，但应展示本局目标、当前分数、步数和最大方块，帮助玩家理解“继续挑战”的上下文。

控件：

- 默认按钮安静，hover 和 focus 才增强。
- focus 要比 hover 更清楚，方便键盘和手柄导航。
- pressed 反馈要短、明确，不要像 selected 状态。
- 禁用态降低透明度和对比，不改变布局尺寸。
- 控件圆角保持一致，默认不超过 8px，除非已有设计系统要求。

## 动效

动效只做三件事：确认输入、解释状态变化、帮助玩家看懂棋盘。

时间范围：

- hover：`0.08s` 到 `0.14s`。
- pressed：`0.04s` 到 `0.08s`。
- 列表或子项 reveal：`0.14s` 到 `0.22s`。
- 面板 intro：`0.20s` 到 `0.30s`。
- 棋盘移动、合并、生成和撤销：以清晰手感为主，不做拖沓表演。

行为规则：

- 同一元素不要同时叠加大位移、大缩放、强发光和长淡入。
- Container 管理的子控件不能被动效改写最终位置。
- 隐藏子控件不播放 reveal。
- 动画不能成为状态可读性的唯一来源。

## 测试与验证

已存在的视觉相关测试：

- `test_visual_polish.gd` 验证背景 shader 加载、Tile 平面色块、主题色不被表现层覆盖、方块文字对比度、UI 动效基础行为和棋盘反馈节点。

后续视觉改动至少检查：

- `docs/VISUAL_STYLE.md` 是否仍描述当前方向。
- `resources/shaders/game_background.gdshader` 的颗粒、扫描线、glow 参数是否保持低刺激。
- `resources/themes/tile_schemes/*.tres` 是否仍由资源定义方块色。
- `GameUiMotionUtility` 的默认态、hover、focus、pressed 是否仍有明确区别。
- 安全 GUT 是否通过。

2. 背景 shader 已较克制，但仍需要在真实截图下确认颗粒不会变成马赛克。

4. 视觉规范目前先以文档和测试固定方向，下一步再逐步推动资源和场景落地。
